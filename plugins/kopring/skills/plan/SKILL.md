# plan — 요구사항 명확화 + 계획서 작성

이 스킬의 산출물(`docs/yyyy-mm-dd-{featName}.md`) 하나만으로 후속 세션이 작업을 이어갈 수 있어야 합니다. 결정·미결·메타·작업 분해를 모두 이 파일에 모읍니다.

코드 수정은 하지 않습니다. 후속 `execute`/`pr` 스킬의 책임입니다.

---

## Step 1. 입력 수집

사용자 입력 형태에 맞춰 자료를 가져옵니다.

| 입력 | 도구 |
|------|------|
| 인라인 텍스트 | 그대로 사용 |
| 로컬 파일 경로 | `Read` |
| Google Docs/Drive (`docs.google.com`, `drive.google.com`) | Google Drive MCP |
| Confluence (`confluence.`, `/wiki/`) | Atlassian MCP |
| Jira 티켓 — **사용자가 명시적으로 요청한 경우에만** | Atlassian MCP |
| 일반 URL / Notion | `WebFetch` |

MCP 인증이 필요하면 사용자에게 인증을 먼저 요청합니다.

---

## Step 2. 메타 정보 수집

다음 3개를 메인 컨텍스트에서 한 번에 정리합니다 (이후 단계에서 재질문 안 함).

1. **베이스 브랜치** — 기본값은 `git branch --show-current` 결과. 다른 브랜치에 PR을 올릴 거면 사용자에게 묻습니다.
2. **지라 키** — 베이스 브랜치명에서 `[A-Z]+-\d+` 패턴 추출 (예: `feat/PROJ-1` → `PROJ-1`). 추출 불가 시 사용자에게 직접 묻습니다.
3. **지라 도메인** — 환경에 캐시(`~/.claude/kopring-jira-domain` 또는 사용자 GLOBAL CLAUDE.md)된 값이 있으면 재사용. 없으면 한 번 묻고 다음 plan 작성 시 자동 사용되도록 사용자에게 어디에 저장할지(`~/.claude/CLAUDE.md` 권장) 안내합니다.

---

## Step 3. Scope 검증

여러 기능이 한 요청에 섞이면 색션 분할이 망가지므로 먼저 확인합니다. 다음 중 하나라도 해당하면 분해 제안:

- 독립 API 2개 이상
- 신규 도메인 동시 생성
- 한 장에 담기 어려운 정책/케이스 다수

해당 시 발견된 기능 목록을 제시하고 어느 것부터 진행할지 묻습니다. 선택된 하나만 Step 4로 갑니다.

---

## Step 4. 요구사항 명확화

좋은 계획서는 개발자가 추측 없이 시작할 만큼의 정보를 담아야 합니다. 검토 영역과 질문 패턴은 `references/clarification-areas.md`를 참고합니다.

질문 원칙:

- **한 번에 하나씩** 묻고 답을 받은 뒤 다음으로 넘어갑니다.
- 답이 정해진 범위 안이면 자유 입력 대신 **다지선다**로 제시 (모호함 ↓, 답변 속도 ↑).
- 합리적 선택지가 여럿이면 trade-off 정리해 추천과 함께 제시 (옵션 2~3개).
- "모름/미정"이면 미결 사항으로 기록하고 진행. 추측해서 채우지 않습니다.
- 코드만 봐도 알 수 있는 것은 묻지 않습니다 (Step 5에서 코드 확인).

---

## Step 5. 코드베이스 매핑 (subagent 위임)

영향 범위와 변경 파일 목록을 정리합니다. **메인 컨텍스트 보호를 위해 subagent에 위임**합니다 — 메인 스레드는 결과 요약만 받습니다.

`Explore` 서브에이전트(또는 `general-purpose`, 단순 탐색이면 `haiku` 모델)에게 다음을 요청:

- 신규 파일: 어떤 패키지/클래스명
- 수정 파일: 어느 클래스의 어느 메서드/필드
- 재사용할 기존 코드: 비슷한 컨트롤러/서비스/유틸
- 데이터 모델 변경: 엔티티/컬럼/인덱스/연관관계
- 외부 의존: 새로 호출할 API/큐/캐시

서브에이전트에게는 **결과를 요약 형식으로만 반환**하도록 지시합니다 (코드 본문 X). 결정이 필요한 부분은 미결 사항으로 회수합니다.

---

## Step 6. 색션(=PR 단위) 분할

작업 크기에 따라 색션을 나눕니다. 각 색션은 **단독으로 빌드/테스트 통과 가능하고 독립 PR로 머지 가능**해야 합니다.

색션 크기 기준:

- 한 색션 = 한 PR. 일반적으로 변경 파일 5개 이내, 단일 책임.
- 여러 레이어를 한 색션에 묶을 수 있지만 의존성이 강한 것끼리만.
- 의존 관계가 있으면 색션 순서로 강제 (Section 1 → 2 → 3).
- 작업이 작으면 색션 1개만으로도 OK.

각 색션 브랜치명 규칙: 베이스가 `feat/PROJ-1`이면 sub는 `feat/PROJ-1-1`, `-2`, `-3` 순. 색션 분할이 1개뿐이면 sub-branch 없이 베이스 브랜치 자체에서 PR.

---

## Step 7. 계획서 작성

`references/plan-template.md`를 `Read`해 템플릿을 가져온 뒤 사용자 답변·코드 매핑·색션 분할을 채워 `docs/{yyyy-mm-dd}-{featName}.md`에 저장합니다.

- `yyyy-mm-dd`: 오늘 날짜
- `featName`: 영어 kebab-case (예: `order-creation`)
- `docs/` 없으면 생성

작성 원칙:

1. 모든 작업 항목은 체크박스(`- [ ]`).
2. 파일 경로/메서드는 구체적으로 (`src/.../OrderService.kt#createOrder`).
3. 미결 사항은 추측해서 채우지 말고 별도 섹션에 기록.
4. 메타 헤더(베이스 브랜치, 지라 키/링크) 반드시 포함 — 후속 스킬이 이걸 읽음.

---

## Step 8. 리뷰 위임 안내

작성 완료 후 사용자에게 경로를 알리고 `review` 스킬 사용을 제안합니다:

```
docs/2026-05-02-order-creation.md 를 작성했습니다.

다음 단계:
- `review` 스킬로 새 세션에서 계획을 검토할 수 있습니다 (권장).
- 검토 후 수정사항이 나오면 같은 파일을 갱신합니다.
- 확정되면 `execute` 스킬로 색션 1부터 구현을 시작합니다.
```

자동으로 `execute`로 넘어가지 않습니다. 사용자가 명시적으로 요청해야 합니다.
