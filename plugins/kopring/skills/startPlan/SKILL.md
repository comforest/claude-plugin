# startPlan — 요구사항 명확화 + 계획서 작성

이 스킬의 산출물(`docs/yyyy-mm-dd-{featName}.md`) 하나만으로 후속 세션이 작업을 이어갈 수 있어야 합니다. 결정·미결·메타·작업 분해를 모두 이 파일에 모읍니다.

코드 수정은 하지 않습니다. 후속 `execute`/`pr` 스킬의 책임입니다.

---

## Step 1. 입력 수집

사용자 입력 형태에 맞춰 자료를 가져옵니다.

| 입력 | 도구 |
|------|------|
| 인라인 텍스트 | 그대로 사용 |
| 로컬 파일 경로 | `Read` |
| Google Docs/Drive (`docs.google.com`, `drive.google.com`) | Google Drive MCP |
| Confluence (`confluence.`, `/wiki/`) | Atlassian MCP |
| Jira 티켓 — **사용자가 명시적으로 요청한 경우에만** | Atlassian MCP |
| 일반 URL / Notion | `WebFetch` |

MCP 인증이 필요하면 사용자에게 인증을 먼저 요청합니다.

---

## Step 2. 메타 정보 수집

다음 4개를 메인 컨텍스트에서 한 번에 정리합니다 (이후 단계에서 재질문 안 함).

1. **통합 브랜치** (integration) — 색션 sub-branch들이 쌓이는 stack base. 기본값은 `git branch --show-current` 결과. 사용자가 `main`/`develop`에 직접 작업 중이라면 새 feature 브랜치(`feat/<지라키>`)로 옮길지 묻습니다.
2. **머지 대상** (merge target) — 통합 브랜치가 최종 머지될 브랜치. 기본값 `main` (또는 프로젝트 컨벤션). 사용자에게 한 번 확인. 색션 1개일 때 PR `--base`로도 사용됩니다.
3. **지라 키** — 통합 브랜치명에서 `[A-Z]+-\d+` 패턴 추출 (예: `feat/PROJ-1` → `PROJ-1`). 추출 불가 시 사용자에게 직접 묻습니다.
4. **지라 도메인** — `~/.claude/CLAUDE.md`를 `Read`로 검사해 `KOPRING_JIRA_DOMAIN: <https://your-org.atlassian.net>` 줄이 있으면 재사용. 없으면 한 번 묻고, 사용자에게 위 형식의 한 줄을 `~/.claude/CLAUDE.md`에 추가해두라고 안내합니다 (다음 plan부터 자동 로드).

---

## Step 3. Scope 검증

여러 기능이 한 요청에 섞이면 색션 분할이 망가지므로 먼저 확인합니다. 다음 중 하나라도 해당하면 분해 제안:

- 독립 API 2개 이상
- 신규 도메인 동시 생성
- 한 장에 담기 어려운 정책/케이스 다수

해당 시 발견된 기능 목록을 제시하고 어느 것부터 진행할지 묻습니다. 선택된 하나만 Step 4로 갑니다.

---

## Step 4. 요구사항 명확화

좋은 계획서는 개발자가 추측 없이 시작할 만큼의 정보를 담아야 합니다. 검토 영역과 질문 패턴은 `references/clarification-areas.md`를 참고합니다.

질문 원칙:

- **한 번에 하나씩** 묻고 답을 받은 뒤 다음으로 넘어갑니다.
- 답이 정해진 범위 안이면 자유 입력 대신 **다지선다**로 제시 (모호함 ↓, 답변 속도 ↑).
- 합리적 선택지가 여럿이면 trade-off 정리해 추천과 함께 제시 (옵션 2~3개).
- "모름/미정"이면 미결 사항으로 기록하고 진행. 추측해서 채우지 않습니다.
- 코드만 봐도 알 수 있는 것은 묻지 않습니다 (Step 5에서 코드 확인).

---

## Step 5. 코드베이스 매핑 (subagent 위임)

영향 범위와 변경 파일 목록을 정리합니다. **메인 컨텍스트 보호를 위해 subagent에 위임**합니다 — 메인 스레드는 결과 요약만 받습니다.

- 단순 파일 위치/심볼 검색이면 `Explore` 서브에이전트.
- 분석/판단이 섞이면 `general-purpose` (가벼운 매핑은 `haiku`, 복잡한 영향도 분석은 `sonnet`).

서브에이전트에게 다음을 요청:

- 신규 파일: 어떤 패키지/클래스명
- 수정 파일: 어느 클래스의 어느 메서드/필드
- 재사용할 기존 코드: 비슷한 컨트롤러/서비스/유틸
- 데이터 모델 변경: 엔티티/컬럼/인덱스/연관관계
- 외부 의존: 새로 호출할 API/큐/캐시

서브에이전트에게는 **결과를 요약 형식으로만 반환**하도록 지시합니다 (코드 본문 X). 결정이 필요한 부분은 미결 사항으로 회수합니다.

---

## Step 6. 색션(=PR 단위) 분할

작업 크기에 따라 색션을 나눕니다. 각 색션은 **단독으로 빌드/테스트 통과 가능하고 독립 PR로 머지 가능**해야 합니다.

색션 크기 기준:

- 한 색션 = 한 PR. 일반적으로 변경 파일 5개 이내, 단일 책임.
- 여러 레이어를 한 색션에 묶을 수 있지만 의존성이 강한 것끼리만.
- 의존 관계가 있으면 색션 순서로 강제 (Section 1 → 2 → 3).
- 작업이 작으면 색션 1개만으로도 OK.

각 색션 브랜치명 규칙: 통합 브랜치가 `feat/PROJ-1`이면 sub는 `feat/PROJ-1-1`, `-2`, `-3` 순.

- **색션 ≥ 2 (stacked)**: 각 색션 PR `--base`는 통합 브랜치, `--head`는 sub-branch. 모든 색션이 통합 브랜치로 머지된 뒤 통합 브랜치를 머지 대상으로 별도 PR(또는 자동 머지된 상태로 정리).
- **색션 = 1**: sub-branch 만들지 않음. 통합 브랜치 자체가 PR `--head`, `--base`는 머지 대상.

---

## Step 7. 계획서 작성 (또는 재작성)

신규 작성과 리뷰 후 재작성을 같은 단계로 처리합니다.

**신규 작성**: `references/plan-template.md`를 `Read`해 템플릿을 가져온 뒤 사용자 답변·코드 매핑·색션 분할을 채워 `docs/yyyy-mm-dd-{featName}.md`에 저장.

- `yyyy-mm-dd`: 오늘 날짜
- `featName`: 영어 kebab-case (예: `order-creation`)
- `docs/` 없으면 생성

**재작성 (리뷰 후 갱신)**: 사용자가 기존 plan과 review 결과를 들고 다시 `startPlan`을 호출한 경우.
1. plan을 `Read` → 메타 + 본문 + `## 리뷰 결과` 모두 확인.
2. `## 리뷰 결과`의 Critical/Major 항목을 가지고 사용자와 한 번에 정리 (Step 4 질문 원칙 적용).
3. 본문(요구사항/색션/코드 매핑)을 `Edit`로 갱신. 메타의 `상태:`는 `drafting`으로 되돌림 (`reviewed` → `drafting`) — 다음 `review`가 다시 통과시켜야 함.
4. `## 리뷰 결과` 섹션은 보존 (다음 라운드의 `### 라운드 N` 비교용).

작성 원칙:

1. 모든 작업 항목은 체크박스(`- [ ]`).
2. 파일 경로/메서드는 구체적으로 (`src/.../OrderService.kt#createOrder`).
3. 미결 사항은 추측해서 채우지 말고 별도 섹션에 기록.
4. 메타 헤더(통합 브랜치, 머지 대상, 지라 키/링크, 상태) 반드시 포함 — 후속 스킬(`execute`/`pr`)이 이걸 읽어 동작합니다.

---

## Step 8. 자동 체이닝 — review 호출

작성 완료 후 사용자에게 경로를 한 줄 알린 뒤 **즉시 `Skill` 도구로 `review`를 plan 모드로 호출**합니다.

```
docs/2026-05-02-order-creation.md 작성 완료. 자동으로 review → execute → pr 단계까지 진행합니다.
```

호출 형식:

```
Skill: review
args: plan <docs/yyyy-mm-dd-{featName}.md>
```

자동 체인 정지 조건은 **`review`가 본문에서 직접 처리**합니다 (Critical/Major 발견 시 중단). 따라서 `startPlan`은 항상 `review` 호출까지 진행하고 종료합니다.

사용자가 자동 진행을 원치 않는다면 plan 작성 시점에 "review/execute 자동 진행 끄고 plan만"이라고 지시할 수 있고, 그 경우 본 Step을 건너뛰고 안내 메시지만 출력합니다.
