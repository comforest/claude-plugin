# 요구사항 명확화 — 검토 영역

`startPlan` Step 4에서 사용자에게 물어 채워야 하는 영역입니다.
정해진 순서나 질문 리스트가 아니라 **입력에서 빠지거나 모호한 부분**을 식별해 묻습니다.

## 영역 목록

- **문제 정의** — 이 기능이 왜 필요한가? 어떤 문제를 해결하나?
- **사용자와 맥락** — 누가, 어떤 상황에서 사용하는가?
- **핵심 동작** — 사용자 관점의 흐름. 결과로 어떤 상태가 되어야 하는가?
- **비즈니스 규칙** — 적용해야 할 정책, 조건, 제약
- **비기능 요구사항** — 성능, 동시성, 트랜잭션 경계, 데이터 정합성, 외부 시스템 연동 실패 정책. Spring Boot에서 누락되기 쉬운 영역이라 명시적으로 다룹니다.
- **예외/에지 케이스** — 비정상 상황의 비즈니스적 처리
- **기대 효과 / 성공 기준** — 잘 동작했다고 판단하는 기준 (가능하면 측정 가능 형태)
- **영향 범위** — 기존 기능/다른 도메인/외부 시스템/운영 도구
- **권한과 사용 범위** — 누가 사용 가능한가? 제한 조건
- **제외 범위 (Out of Scope)** — 이번 작업에서 명시적으로 빼는 것

## 다지선다 예시

답이 정해진 범위 안에 있을 때 자유 입력 대신 사용:

- 권한: (1) 인증 불필요 (2) 로그인 사용자 (3) 특정 Role (4) 기타
- DB 변경: (1) 없음 (2) 컬럼 추가 (3) 신규 테이블 (4) 기존 컬럼 수정
- 트랜잭션 경계: (1) 한 트랜잭션 (2) 분리 + 보상 트랜잭션 (3) 결과적 일관성

## 대안 탐색이 필요한 신호

단순 질문 대신 옵션 비교를 제시할 타이밍:

- 비즈니스 정책에 분명한 정답이 없음 (재고 0일 때 차단 vs 백오더 vs 알림 후 차단)
- 같은 목표를 다른 방식으로 달성 가능 (즉시 처리 vs 비동기 큐)
- 사용자 답변이 다른 영역에 파급되는데 본인이 그 연결을 인지 못 한 것으로 보일 때

제시 형식:

```
이 부분은 합리적 선택지가 여럿이라 짚고 갑니다.

옵션 A — [이름]
  장점: ...
  단점: ...

옵션 B — [이름]
  장점: ...
  단점: ...

(추천: B — 이유)
어떤 방향으로 갈까요?
```

옵션은 2~3개까지만. 사용자가 이미 명확히 정한 사항에는 대안을 제시하지 않습니다 (불필요한 흔들기 방지).
