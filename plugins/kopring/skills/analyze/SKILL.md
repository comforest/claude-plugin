# analyze — 기획 검토 스킬

사용자가 제공한 짧은 텍스트나 기획 문서를 분석하여 **구현 계획 전에 기획 수준의 검토서**를 작성합니다.
기술 구현 방법이 아닌 *무엇을, 왜, 어떤 범위로* 만들지를 명확히 하는 것이 목표입니다.
부족한 정보는 사용자에게 하나씩 질문하여 채우고, 확인된 내용을 `docs/yyyy-mm-dd-{featName}.md`에 저장합니다.

---

## Step 1. 코드베이스 컨텍스트 파악

분석을 시작하기 전에 관련 코드베이스를 **짧게** 살펴 기존 구조를 파악합니다.
이 단계의 목적은 두 가지입니다.

1. 코드만 봐도 알 수 있는 것은 사용자에게 묻지 않기 (질문 횟수 절감)
2. 영향 범위와 기존 컨벤션을 정확히 짚어 명세서 품질을 높이기

요청에서 키워드(도메인명, API명, 엔티티명 등)를 추출해 다음을 확인합니다.

- **유사 / 관련 기능**: 비슷한 API · 도메인 · 유스케이스가 이미 있는가? 있다면 어떤 패턴을 따르는가?
- **공통 인프라**: 인증 / 인가 방식, 에러 응답 포맷, 트랜잭션 전략, 공통 예외 처리, 페이지네이션 컨벤션
- **데이터 구조**: 영향 받을 가능성이 있는 엔티티 · 테이블, 기존 컬럼 구성, 연관관계
- **연관 도메인**: 기능이 닿을 수 있는 다른 도메인(예: 이벤트 발행, 알림, 정산)

도구 사용 가이드:

- `Grep` · `Glob`로 키워드 기반 빠른 탐색을 먼저 합니다.
- 의심되는 **핵심 파일만** `Read`합니다. 디렉토리 트리 전체를 훑지 않습니다.
- 새 기능이라 관련 코드가 거의 없으면 "관련 코드 없음"이라고 짧게 확인하고 넘어갑니다.

이 단계는 **1~3분 안에 끝나야 합니다**. 더 깊은 코드 분석은 구현 단계의 몫이며, 분석 단계에서는 "어디에 영향이 있는지"와 "코드만으로 답할 수 있는 질문이 무엇인지"를 식별하는 정도로 충분합니다. 길어진다 싶으면 멈추고 Step 2로 진행하세요.

파악한 내용은 메모로 들고 다음 단계에 활용합니다. 이미 코드에서 확정된 사실(공통 에러 형식, 기본 인증 방식 등)은 Step 4에서 다시 묻지 않습니다.

---

## Step 2. 입력 읽기

사용자 입력 형태에 따라 적절한 도구로 내용을 가져옵니다.

- **텍스트 입력**: 기능 설명이 인라인으로 주어진 경우 → 그대로 사용
- **파일 경로 입력**: 로컬 파일 경로가 주어진 경우 → Read로 문서 내용을 읽음
- **링크 입력**: URL이 주어진 경우 → 아래 기준으로 도구 선택

| 링크 유형 | 판단 기준 | 사용 도구 |
|-----------|-----------|-----------|
| Google Docs / Google Drive | `docs.google.com` 또는 `drive.google.com` 포함 | Google Drive MCP |
| Confluence | `confluence.` 또는 `/wiki/` 포함 | Atlassian MCP |
| 일반 웹 페이지 / Notion / 기타 | 그 외 모든 URL | WebFetch |

MCP 사용 시 인증이 필요하면 사용자에게 인증을 요청한 뒤 진행합니다.

---

## Step 3. Scope 검증

분석을 시작하기 전에 **요청 범위가 단일 기능으로 처리 가능한지** 판단합니다.
한 명세서에 여러 기능이 섞이면 누락이 생기고 검토가 어려워지기 때문입니다.

아래 중 하나라도 해당하면 scope 과다로 판단합니다:

- 독립적인 API가 2개 이상 포함 (예: "주문 API + 결제 API + 알림 API")
- 서로 다른 도메인을 동시에 신규 생성 (예: "회원 도메인이랑 상품 도메인 같이 만들어줘")
- 명세서 한 장에 담기 어려울 정도로 정책/케이스가 복잡하게 나열된 경우

scope 과다로 판단되면 분석을 중단하고 사용자에게 분해를 제안합니다:

```
요청에 독립적인 기능이 여러 개 포함되어 있습니다.
한 번에 분석하면 명세서가 복잡해지고 누락이 생길 수 있어 순서대로 나눠 진행하는 것을 권장합니다.

확인된 기능 목록:
1. [기능 A] — [간단한 설명]
2. [기능 B] — [간단한 설명]
3. [기능 C] — [간단한 설명]

어떤 기능부터 분석을 시작할까요?
```

사용자가 선택하면 해당 기능 하나만 대상으로 Step 4부터 진행합니다.

---

## Step 4. 부족한 정보 채우기

좋은 요구 사항 명세서는 **개발자가 추측 없이 구현을 시작할 수 있을 만큼**의 정보를 담아야 합니다.
이를 위해 아래 영역들이 모두 사용자 답변(또는 명시적 미결 처리)에 근거해야 합니다.
정해진 질문 리스트를 따르는 게 아니라, **이 영역 중 입력에서 빠져 있거나 모호한 부분**을 식별해 묻습니다.

### 검토에 필요한 영역

- **문제 정의** — 이 기능이 왜 필요한가? 어떤 문제를 해결하나?
- **사용자와 맥락** — 누가, 어떤 상황에서 사용하는가?
- **핵심 동작** — 사용자 관점에서 기능이 어떻게 흘러가는가? 결과로 어떤 상태가 되어야 하는가?
- **비즈니스 규칙** — 적용해야 할 정책, 조건, 제약
- **비기능 요구사항** — 성능 · 동시성 · 트랜잭션 경계 · 데이터 정합성 · 외부 시스템 연동 실패 시 정책 등. Spring Boot 맥락에서 누락되기 쉬운 영역이므로 명시적으로 다룹니다 (해당 없으면 그렇게 기록).
- **예외와 에지 케이스** — 비정상 상황에서 비즈니스적으로 어떻게 처리해야 하는가?
- **기대 효과 / 성공 기준** — 이 기능이 잘 동작했다고 판단하는 기준은 무엇인가?
- **영향 범위** — 기존 기능이나 다른 도메인에 어떤 영향을 주는가?
- **권한과 사용 범위** — 누가 사용할 수 있는가? 제한이 있는가?
- **제외 범위** — 이번 작업에서 명시적으로 빼는 것

각 영역에 대해 입력만으로 명확히 알 수 있으면 질문하지 않습니다. 빠지거나 모호하면 묻습니다.
영역 사이의 우선순위는 답변 맥락에 따라 자연스러운 순서로 결정합니다 (보통 상위 영역에서 하위 영역으로 좁혀가는 흐름이 매끄럽습니다).

### 질문 방식

- 한 번에 **질문 하나만** 합니다. 답을 받은 뒤 다음으로 넘어갑니다.
- 답이 정해진 범위 안에 있다고 판단되면 자유 입력 대신 다지선다로 제시합니다 — 답이 더 빠르고 모호함이 줄어듭니다.
  - 예: 권한 → (1) 인증 불필요 (2) 로그인 사용자 (3) 특정 Role (4) 기타
  - 예: DB 변경 → (1) 없음 (2) 컬럼 추가 (3) 신규 테이블 (4) 기존 컬럼 수정
- 답변이 불분명하면 해당 영역에서 재질문합니다. 추측해서 넘어가지 않습니다.
- "모름" 또는 "미정"이면 미결 사항(Open Issues)에 기록하고 다음으로 넘어갑니다.
- 코드베이스에서 이미 확인 가능한 정보(공통 에러 형식, 인증 방식 등)는 묻지 않습니다.

### 대안 탐색

답이 정해진 범위 안에 있지 않고 **합리적인 선택지가 여럿**일 때는, 단순히 묻기 전에 모델이 trade-off를 정리해 제시합니다. 사용자가 한쪽 방향만 막연히 떠올리고 있을 수 있으므로, 전체 선택 폭을 보여주는 것이 목적입니다.

대안 탐색이 필요한 신호:

- 비즈니스 정책에 분명한 정답이 없음 (예: 재고 0일 때 차단 vs 백오더 vs 알림 후 차단)
- 같은 목표를 다른 방식으로 달성 가능 (예: 즉시 처리 vs 비동기 큐)
- 사용자 답변이 다른 영역에 영향을 주는데 본인이 그 연결을 인지 못 한 것으로 보일 때

제시 형태:

```
이 부분은 합리적 선택지가 여럿이라 짚고 갑니다.

옵션 A — [이름]
  장점: ...
  단점: ...

옵션 B — [이름]
  장점: ...
  단점: ...

(추천: B — 이유)
어떤 방향으로 갈까요?
```

원칙:

- **2~3개까지만** 제시합니다. 4개 이상은 의사결정을 어렵게 합니다.
- **추천 옵션과 이유**를 함께 제시하되, 사용자가 다른 방향을 택하면 그대로 따릅니다.
- 사용자가 이미 명확히 정한 사항에는 대안을 제시하지 않습니다 (불필요한 흔들기 방지).

모든 영역이 사용자 답변 또는 미결 사항으로 정리되면 Step 5로 진행합니다.

---

## Step 5. 기획 검토서 작성

`references/spec-template.md`를 Read하여 템플릿을 가져온 뒤, 사용자 답변을 채워 `docs/yyyy-mm-dd-{featName}.md`로 저장합니다.

- `yyyy-mm-dd`: 오늘 날짜 (예: 2026-04-28)
- `featName`: 기능명을 영어 kebab-case로 (예: `order-creation`, `user-profile-update`)
- `docs/` 폴더가 없으면 자동으로 생성합니다.

---

## Step 6. 확인 및 확정

생성된 기획 검토서의 경로를 사용자에게 알리고 검토를 요청합니다:

```
docs/2026-04-28-order-creation.md 를 생성했습니다.

내용을 검토해주시고:
- 수정이 필요한 항목이 있으면 알려주세요.
- 확인이 완료되면 구현 계획(plan) 단계로 넘어갈 수 있습니다.
```

수정 요청이 오면 해당 항목만 반영하여 파일을 업데이트합니다.
미결 사항(Open Issues)이 남아 있어도 사용자가 진행을 원하면 그대로 확정합니다.
