# execute — 색션 단위 구현 스킬

이 스킬은 **plan 계획서를 단일 진실 공급원(SSOT)** 으로 보고 그 안의 한 색션을 끝까지 구현합니다. 진행 중 모든 결정·완료 표시는 plan md에 반영되어, 다음 세션이 plan만 읽고도 이어갈 수 있어야 합니다.

토큰 절약 핵심:

- 한 번에 **한 색션만** 구현. 여러 색션을 한 세션에 끌고 가지 않음 (PR 단위 = 컨텍스트 단위).
- 큰 신규 파일·반복 작업은 subagent에 위임. 메인은 조율과 plan 갱신.
- 메인 스레드가 코드베이스 전체를 읽지 않음 — plan에 적힌 파일·메서드 위주로만.

---

## Step 1. 계획서 식별 및 로드

사용자가 plan 경로를 명시했으면 그 파일, 아니면 `docs/`에서 가장 최근의 `yyyy-mm-dd-*.md`를 후보로 제시. 없으면 "먼저 `startPlan` 스킬로 계획서를 만들어주세요" 안내 후 중단.

`Read`로 계획서 전체를 한 번 로드합니다 (메타·색션·진행 상태). 큰 파일이라도 plan은 보통 수백 줄 수준이라 메인 컨텍스트에 들고 갑니다.

확인할 메타:
- 통합 브랜치 / 머지 대상
- 지라 키 / 링크
- 진행 상태 표 (어느 색션이 `done`/`in-progress`/`pending`/`pr-open`/`merged` 인지)

---

## Step 2. 작업할 색션 결정

진행 상태 표 기준:

- `in-progress` 색션이 있으면 그것 — 중단된 작업 재개로 간주.
- 없으면 `pending` 중 가장 작은 번호.
- 사용자가 특정 색션을 명시했으면 그것 우선.

선택한 색션을 `in-progress`로 표시 (`Edit`로 진행 상태 표 갱신). 이 시점에 plan 메타 헤더의 `상태:`가 `reviewed`거나 `drafting`이면 `in-progress`로 함께 갱신 (전체 상태 책임은 plan-template의 책임 표 참조).

색션의 의존을 확인 — 앞 색션이 머지되어야 진행 가능한 색션이면 사용자에게 "색션 N은 색션 N-1 머지 후 진행해야 합니다. 그래도 진행할까요?"라고 묻습니다.

---

## Step 3. 작업 브랜치 준비

이 시점에 sub-branch로 옮기지 않습니다. **통합 브랜치 위에서 그대로 작업**합니다 — sub-branch 생성은 `pr` 스킬이 PR 직전에 처리.

이유: 작업 도중 sub-branch에 있으면 색션 분할이 잘못됐을 때 되돌리기 번거롭고, 한 색션만 작업하므로 실수로 다른 색션 변경을 섞을 위험이 적습니다.

### 3-1. 통합 브랜치 위치 확인

```bash
git status
git branch --show-current
```

현재 브랜치가 plan 메타의 통합 브랜치와 다르면 사용자에게 보고하고 체크아웃 여부를 확인합니다. 임의로 바꾸지 않습니다.

### 3-2. 머지 동기화 + plan 상태 전이

`pr-open` 색션이 origin에서 머지되었는지 확인하고 plan을 갱신합니다 (`pr` 스킬은 `pr-open`까지만 채우고, `merged` 전이는 이 스킬의 책임).

```bash
git fetch origin
gh pr view <PR_NUMBER> --json state -q .state    # MERGED / OPEN / CLOSED
```

- `MERGED` → 해당 색션 행 상태를 `pr-open` → `merged`로 `Edit`. 모든 색션이 `merged`면 plan 메타 헤더의 `상태:`를 `done`으로 갱신.
- `OPEN`/`CLOSED` → plan 그대로 두고 사용자에게 안내.

머지된 색션이 있었다면 통합 브랜치를 원격과 동기화:

```bash
git pull --ff-only origin "$INTEGRATION_BRANCH"
```

fast-forward가 안 되면(rebase·squash 머지로 히스토리가 갈라짐) 사용자에게 보고하고 처리 방향(`reset --hard origin/$INTEGRATION_BRANCH` 등)을 확인. 임의 reset 금지.

### 3-3. 워킹 트리 확인 + stash 추적

`git status`가 깨끗하지 않으면 사용자에게 처리 방식을 묻습니다 — `stash` 또는 별도 브랜치로 분리 커밋. **"무시하고 진행"은 다른 색션 변경이 색션 N에 섞일 위험이 있어 옵션에서 제외합니다.**

`stash`를 선택했다면 plan 메타에 한 줄 메모를 남깁니다 (다음 세션이 잊지 않도록):

```
> stash: <stash@{0} message> — 색션 N 종료 후 pop 필요
```

색션 종료(Step 5) 시 메모를 보고 `git stash list`로 확인, 사용자에게 pop 여부를 묻습니다.

---

## Step 4. 작업 항목 순차 처리

색션의 체크박스 항목을 위에서 아래로 처리합니다.

각 항목마다:

1. **구현 방식 선택**
   - 단일 파일 수정·작은 신규 파일 → 메인이 직접 (Read/Edit/Write)
   - 신규 파일 여럿·반복 패턴·테스트 다수 → `Agent`(subagent) 위임. 모델은 작업 복잡도에 따라 `haiku`(보일러플레이트) ~ `sonnet`(일반) ~ `opus`(설계 결정 포함).

2. **subagent 위임 시** 다음을 명시:
   - 정확한 파일 경로와 시그니처
   - 프로젝트 컨벤션 (subagent에게 비슷한 기존 파일 한두 개를 참조하도록 지시)
   - 테스트도 같이 만들지 여부
   - 결과 형식 (부분 실패 회수 가능하도록 강제):
     ```
     성공: <경로 목록>
     실패: <경로 + 사유> (없으면 줄 자체 생략)
     ```
   - 코드 본문은 회수하지 않음.

   subagent가 부분 실패로 돌아오면 **성공한 항목만 체크박스 갱신**하고, 실패 항목은 plan에 남겨둔 채 사용자에게 보고. 다음 세션도 plan만 보면 어디서 멈췄는지 알 수 있음.

3. **구현 직후** 해당 항목의 체크박스를 `[x]`로 갱신:
   ```
   - [ ] OrderService#createOrder → - [x] OrderService#createOrder
   ```
   `Edit`로 plan md를 즉시 업데이트. 한꺼번에 몰아서 갱신하지 않음 — 중도 중단 시 정확한 진행도가 남아야 함.

4. **테스트 실행** — 항목이 로직 변경이면 관련 테스트만:
   ```bash
   ./gradlew test --tests <FQCN>
   ```
   실패하면 수정 후 재실행. 항목 단위로 통과를 확인한 뒤 다음으로.

---

## Step 5. 색션 완료 처리

색션의 모든 체크박스가 `[x]`가 되면:

1. **전체 빌드 + 테스트** — `./gradlew build` (Windows: `gradlew.bat build`). `build`가 test를 포함하므로 한 번이면 충분. 실패 시 색션 안에서만 수정, 다른 색션 코드는 건드리지 않음.
2. **plan 진행 상태 표 갱신** — 색션 상태 `in-progress` → `done`. PR 칸은 `pr` 스킬이 채울 자리이므로 `-` 유지.
3. **변경 사항 확인** — `git status` + `git diff --stat`. 계획에 없는 파일이 변경되었으면 사용자에게 보고하고 의도된 변경인지 확인. 의도된 변경으로 결정되면 **plan의 색션 작업 항목·변경 파일 목록을 `Edit`로 추가**한 뒤 진행 — plan만 보고 이어가야 한다는 SSOT 원칙 강제.

4. **stash 회수** — Step 3-3에서 stash를 만들었다면 메타의 stash 메모를 확인하고 사용자에게 `pop` 여부를 묻습니다. 처리 후 메타 라인 제거.

---

## Step 6. 자동 체이닝 — pr 호출

색션이 완료되면 사용자에게 한 줄 요약을 출력한 뒤 **즉시 `Skill` 도구로 `pr`을 호출**합니다.

```
색션 N 완료 (변경 파일 X개, 테스트 통과). pr 자동 진행.
```

호출 형식:

```
Skill: pr
args: <plan 파일 경로> <색션 번호>
```

색션 코드 검토를 사용자에게 보여줘야 한다면(예: 위험 변경 또는 plan 외 변경 발생), 자동 체인을 중단하고 사용자 결정 대기. 그 외에는 항상 pr까지 자동 진행.

---

## 주의사항

- **plan을 권한의 원천으로**: plan에 없는 작업·파일·결정은 임의로 추가하지 않습니다. 필요하면 멈추고 사용자에게 plan 갱신 여부를 묻습니다.
- **체크박스 즉시 갱신**: 한 항목 완료 → 즉시 plan md 갱신. 배치로 미루면 중도 중단 시 진행도가 어긋납니다.
- **색션 간 코드 섞기 금지**: 한 세션 = 한 색션 = 한 PR. 색션 2 작업이 색션 1 파일을 살짝 바꾸고 싶어져도, 따로 색션으로 분리하거나 plan을 갱신해 명시한 뒤 진행.
- **커밋은 색션 끝에만 (Conventional Commits)**: 색션 중간 커밋은 선택. 마지막 커밋 메시지에 지라 키 포함:
  ```
  feat(order): create order endpoint

  - 구현한 작업 항목 1
  - 구현한 작업 항목 2

  Refs: PROJ-1
  ```
- **Windows 환경**: `./gradlew` 대신 `gradlew.bat` 사용.
